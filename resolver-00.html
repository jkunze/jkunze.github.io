The resolver.txt Collection Description File
draft-kunze-resolver-00

Abstract

This document specifies /resolver.txt, a simple convention for a URL path and a set of YAML-encoded elements. The elements describe a server (resolver) that responds to URLs containing identifiers assigned to collection objects.

1. Introduction

Across the sciences and the humanities, scholars have traditions of collecting, curating, and citing research objects. Discipline-specific, web-based collections (databases) have emerged, some of which are now indispensable parts of the research infrastructure; for example, GenBank [GenBank] in biomedicine. There are many such collections -- over 500 in life sciences alone -- and in an increasingly inter-disciplinary world, it can be useful to leverage them with machine-readable and harvestable descriptions.

Collections often have schemes (systems) of unique identifiers. While schemes vary, the identifiers tend to be short strings (5-20 characters) and appear in scholary citations, and they only become actionable when appended to a collection-specific base URL. There are thousands of such collections, and cross-collection linking requires bookmarking fragile URLs with embedded identifiers.

To make such identifiers more robust while preserving their compact appearance, it would be useful to provide a well-known base URL that can redirect a scheme plus short identifier string to a scheme-specific (collection-specific) service.

We define a "collection resolver" (or just "resolver") to be a web service that accepts URLs with embedded collection identifiers and responds with such things as object access and metadata. By "meta-resolver" we mean a web server that can recognize enough about an incoming URL and redirect to one or more collection resolvers.

2. The /resolver.txt file

Here we specify an optional file that holds one or more collection descriptions.

2.1. Location

By convention, on any given web server the file resides at the URL path, "/resolver.txt", analogous to robots.txt [robots.txt]. This file is entirely optional and would only exist if a server administrator wished to expose it to the world. The URL for the file is constructed by appending the above path to the hostname (and optional port), so for the server at www.example.com, the URL to access the file would be 

    http://www.example.com/resolver.txt

2.2. Format

The file is made machine readable using YAML [YAML]. Here is an example file containing one collection description. 

   prefix: pmid
   access pattern: www.ncbi.nlm.nih.gov/pubmed/
   full name: PubMed
   contact: <admin@example.org>
   description: PubMed is a service of the U.S. National Library of
      Medicine that includes citations from MEDLINE and other life
      science journals for biomedical articles back to the 1950s.
   id pattern: ^\d+$
   id normalization: RH        # removes any hyphens before resolving
   test id: http://www.ncbi.nlm.nih.gov/pubmed/16333295

The general form of a collection description follows. If there is more than one description, separate each with the YAML "---" delimiter. 

   prefix: <unique string registered manually (next section)>
   access pattern: <URL path with optional embedded $id>
     # if $id doesn't appear in the pattern, it will be appended
     # prepend http(s) if provider requires, else retain user choice
   contact: <email address>
   full name: <description of scheme/collection/shoulder>
   description: <prose summary of scheme/collection and provider(s)>
   id pattern: <regular expression, Perl compatible (pcre.org)>
   id normalization: <string containing one or more of
        RH    remove hyphens
        2U    convert to upper case
        2L    convert to lower case
        4E    hex-encode (every 4 bits)
        6E    base64-encode (every 6 bits)>
   test id: <URL containing a test id, may be preceded by "HEAD ">
     # precede by HEAD if provider prefers it over a GET method test

3. Meta-resolver support: prefix uniqueness and harvest control

This section describes one way to support a meta-resolver. It requires that each collection be associated with an identifier scheme name -- the prefix string above -- that is unique among all scheme names known to the meta-resolver. A registry mirrored at the following locations 

     http://xxx.example.org/schemes.txt
     http://yyy.example.com/schemes.txt

is used to maintain uniqueness. It is a plain text file, each line of which contains a scheme name optionally followed by a URL from which to harvest a resolver.txt file. If no URL is given, harvesting is suspended for that scheme. Comment lines are introduced by '#'. For example, 

     foo http://zzz.example.net/resolver.txt
     bar http://zzz.example.net/resolver.txt
     zaf
     #foo http://www.example.com/resolver.txt   old location of foo

shows three unique schemes, two of them with active harvesting. To signal your desire to be registered and subject to daily harvesting, send an email to XXX@example.org with your proposed scheme name (prefix) and the location of your resolver.txt file. Use that address to request any other changes in the scheme name, resolver.txt location, or to turn off harvesting. [I guess with that URL, resolver.txt need not always be at the root of the web server --jak]

Once you have obtained an approved scheme name that is unique across the registry, use that name henceforth as the value of the prefix in your resolver.txt file. For security reasons, while harvesting is active, any harvested resolver.txt file changes containing a scheme name differing from the one currently registered MUST be rejected as an exception. Thus any changes to a registered prefix are to be negotiated by email.
