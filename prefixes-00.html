Prefix-Based Collection Access
draft-kunze-prefixes-00

Abstract

This document specifies a YAML [YAML] file that serves as an open registry of unique collection prefixes. These prefixes can be used by meta-resolvers to redirect identifiers to appropriate collection resolvers.

1. Introduction

Across the sciences and the humanities, scholars have traditions of collecting, curating, and citing research objects. These include articles, data files, tables, table rows, software, physical specimens, events, and vocabulary terms. Discipline-specific, web-based object collections (eg, databases) have emerged, some of which are now indispensable parts of the research infrastructure; for example, GenBank [GenBank] in biomedicine. There are many such collections -- over 500 in the life sciences alone -- and in an increasingly inter-disciplinary world, it can be useful to create common conventions for accessing objects across collections.

It often not obvious how to take object identifiers that are locally unique within a collection and derive globally unique identifiers that are actionable from a web browser. Each collection has its own scheme (syntax and assignment conventions) for its locally unique identifiers (LUIs). While schemes vary, LUIs tend to be short strings (5-20 characters) that often appear in citations after a prefix identifying the collection. This sort of collection prefix plus LUI combination is a "compact identifier" familiar inside disciplines that use the given collection. The problem is that it is not easy for people from outside the discipline (researchers, publishers) to gain access to cited objects. There are thousands of collections, each with its own idiosyncratic and evolving way of constructing actionable identifiers (URLs) from LUIs.

We define a "collection resolver" (or just "resolver") to be a web service that accepts URLs with embedded collection identifiers and responds with such things as object access and metadata. By "meta-resolver" we mean a web server that can recognize enough about an incoming URL and redirect to one or more collection resolvers. A meta-resolver can provide a single way to construct URLs from compact identifiers, namely, by appending the prefix plus LUI to the URL basename of the meta-resolver. The meta-resolver that keeps track of the myriad collection resolver access methods can take an incoming resolution request, lookup the appropriate redirection rule (access method) based on the prefix, and transparently forward the request to the correct collection resolver. This shields people from the details of and changes in collection resolver access methods, and makes it easy to associate compact citations with actionable hypertext links.

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119 [RFC2119].

2. Compact identifiers

In this document a "compact identifier" is a string constructed by concatenating a scheme, colon, and LUI (an identifier locally unique in the collection designated by the scheme), for example, 

     pmid:16333295

specifies the LUI, 16333295, in the PubMed collection [two more example lines above would be helpful]. When a collection has more than one provider, it is possible to prepend a short "provider code" and a "/", for example, 

     epmc/pmid:16333295

specifies that the European provider collection resolver is preferred.

In general, a "prefix" is thus either a scheme name or a scheme name preceded by a provider code. The "/" convention makes it easy to surface the primary collection (scheme) designator and forward requests to the requested resolver. In the absence of a provider code, a meta-resolver will forward to a collection resolver of its choice.

3. The "prefixes" file

This section specifies an open "prefixes" registry of unique collection designators. It is maintained as a text file in YAML [YAML] format maintained in a github.com repository. Additions, corrections, and deletions are requested via the github issue tracking mechanism. It is made available for access and download at the URL, 

  https://github.com/prefixcommons/prefixes/blob/master/registry.yaml

The file consists of a sequence of "prefix records" in YAML [YAML] format. There should be a prefix record for each combination of scheme name and provider. Here is an example of three records defining three prefixes (pmid, epmc/pmid, and flybase). 

     scheme: pmid
     - redirect: http://www.ncbi.nlm.nih.gov/pubmed/$id
     - test: 16333295
     - more: http://www.ncbi.nlm.nih.gov/PubMed/
     - admin: dlipman
     scheme: pmid
     - provider: epmc
     - redirect: http://europepmc.org/abstract/MED/$id
     - test: 16333295
     - more: http://europepmc.org/
     - admin: jkunze
     scheme: flybase
     - redirect: http://flybase.org/reports/$id.html
     - test: FBgn0011293
     - normalize: NH, 2U
     - more: http://flybase.org/
     - admin: njuty

The general layout of a "prefix record" is: 

   scheme: <required alphanumeric identifier scheme name>
   - provider: <optional alphanumeric provider code>
   - redirect: <required URL forwarding rule>
   - admin: <required github handle>
   - test: <required test identifier>
   - title: <optional full title of the prefix>
   - normalize: <optional coded transformation rule> [toss?]
   - more: <optional URL for access more information>
   - note: <optional text statement>
   # comment lines introduced with '#' are allowed

User resolution requests arrive at a meta-resolver as a prefix, a colon (":"), and an LUI. As before, the prefix is either just a scheme or a scheme preceded by a provider code. What happens next depends on individual prefix record elements described below.

3.1. Element: scheme

A string of lowercase letters and digits defining the identifier type, typically for a given collection or database. The combination of scheme and provider (below) must be unique across the registry. To indicate a deprecated scheme, end the string with the reserved substring, " - deprecated". This element is required.

3.2. Element: provider

A string of lowercase letters and digits defining one provider for an identifier scheme. The combination of scheme and provider (above) must be unique across the registry. If this element is not present, only one provider is assumed. To indicate a deprecated provider, end the string with the reserved substring, " - deprecated".

3.3. Element: redirect

A rule specified as a URL-like path defining how to forward compact identifiers to a specific provider. If the literal string "$id" appears in the rule, the redirection target will be formed by replacing the "$id" with the LUI. If no "$id" appears, the LUI will be appended. Unless the rule begins with a fixed URI scheme (eg, "http:" or "https:"), the incoming resolution request's choice of URI scheme will be honored (preserved) in the forwarded request. This element is required.

3.4. Element: admin

The login name of the person authorized to make changes for this scheme/provider combination, generally the person who requested it originally. This element is required and may be repeated.

3.5. Element: test

A sample LUI that will be tested before any change to a "redirect" element will be approved (eg, by an automated process). If the result of forwarding the LUI does not ultimately return an HTTP status code in the 200 range, a change in redirect will be denied. This element is required.

3.6. Element: title

A text string containing the full name of the prefix. This element is optional.

3.7. Element: normalize

A rule specified as a coded string with instructions for normalizing LUIs before forwarding. An example string with three codes is "2U, NH,NS", which instructs a meta-resolver to convert to upper case and remove all hyphens and whitespace. The defined codes are 

        2U    convert to upper case
        2L    convert to lower case
        NH    remove hyphens
        NS    remove whitespace
        TR    trim just external whitespace
        SS    squeeze multiple whitespace chars into one space
        4E    hex-encode (every 4 bits)
        6E    base64-encode (every 6 bits)>

This element is optional. [ For simplicity I'm thinking it might be better to remove this element. --jak ]

3.8. Element: more

A URL that leads to a web page with more information about the prefix. If the page contains schema.org tags, the meta-resolver MAY exploit them for descriptive information. This element is optional.

3.9. Element: note

A text string containing arbitrary annotations. This element is optional and may be repeated.
